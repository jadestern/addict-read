## 🚀 Phase 6: UX 개선 및 고급 기능 (다음 우선순위)

### 1. 스마트 UI/UX 개선 ⭐️ **[우선순위]**
- **조건부 앱 설명**: 로그인 상태 + 기사 존재 시 설명 숨김
  - 비로그인 상태 OR 피드/기사 없음 → 앱 설명 표시
  - 로그인 + 기사 있음 → 앱 설명 숨김 (깔끔한 UI)
- **타이틀 중복 제거**: 헤더에 "Feedic" 있으므로 메인 타이틀 조건부 표시

### 2. 기사 상세 페이지 및 내용 관리 ⭐️
- **기사 상세 페이지**: 전체 기사 내용 표시를 위한 별도 페이지/모달
- **기사 전문 저장**: RSS에서 description/content 전체 텍스트 저장
- **기사 미리보기**: 메인 화면에서 제목 + 한 줄 요약 (ellipsis)
- **내부 콘텐츠**: 외부 링크 대신 앱 내에서 기사 읽기

### 3. 간단한 읽음 상태 관리 ⭐️
- **Jazz CoValue 스키마 활용**: `Article` 모델의 `isRead` 필드 적극 활용
- **자동 읽음 처리**: 상세페이지 진입 시 자동으로 읽음 상태 변경
- **전체 읽음 버튼**: 모든 기사를 한 번에 읽음 처리하는 버튼
- **읽음 상태 UI**: 읽은 기사는 시각적으로 구분 (opacity, 색상 등)
- **읽지 않음 기능 제거**: 복잡성 제거, 읽음만 처리

### 4. 성능 및 사용성 최적화
- **기사 개수 제한**: RSS 파서에서 최신 3개만 반환 (성능)
- **링크 아이콘**: 외부 링크를 이모지(🔗)로 표시
- **기사 카드 개선**: 읽음/안읽음에 따른 스타일 차별화

### 5. 간단한 상태 기반 네비게이션
- **컴포넌트 기반 show/hide**: React Router 없이 상태로 페이지 전환
- **상세보기 모드**: `selectedArticle` 상태로 메인/상세 화면 전환
- **뒤로가기 버튼**: 간단한 `setSelectedArticle(null)` 처리
- **부드러운 전환**: CSS transition으로 자연스러운 애니메이션

## 🤔 기술적 고민사항

### 현재 아키텍처 검토

**Jazz CoValue vs 로컬 상태**:
- 현재: Jazz에 URL만 저장, 기사는 로컬 상태
- 고려사항: 기사도 Jazz에 저장할 것인가?
- **권장**: 현재는 로컬 상태로 시작, 추후 Jazz 동기화 고려

**Mock vs 실제 RSS 파싱**:
- Mock 장점: 빠른 개발, 안정적 테스트
- 실제 파싱 고려사항: CORS, 서버사이드 필요
- **권장**: Mock으로 UI/UX 완성 후 실제 파싱 단계적 도입
